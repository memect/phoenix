# AgentScope Agent 架构总结

## 1. 系统概述

AgentScope Agent 是一个多 Agent 协作系统，用于自动化开发和优化数据提取程序。

传统做法中，编写提取程序需要开发者反复阅读文档、理解数据结构、编写代码、运行测试、修正错误——这是一个高度迭代的过程。AgentScope Agent 将这个过程自动化：通过多个 AI Agent 的分工协作，自动完成从业务理解、代码编写到质量验证的完整闭环。

核心价值：将人工数小时的提取程序开发，压缩为一次自动化运行，同时保证代码质量和提取准确率。

## 2. 架构设计

### 2.1 多 Agent 协作架构

系统采用三个 Agent 分工协作的架构，各司其职：

| 角色 | 职责 | 类比 |
|------|------|------|
| **ExtractDevAgent** | 主执行者。负责编写提取代码、运行测试、修复问题 | 一线开发者 |
| **Supervisor** | 监督者。每轮检查进度、评估质量、给出策略指导 | 技术负责人 |
| **BusinessAgent** | 业务顾问。分析数据集语义，生成业务理解文档，回答业务问题 | 领域专家 |

这种分离的设计意图是：让写代码的 Agent 专注于编码，让监督者从全局视角把控方向，让业务顾问提供领域知识——避免单个 Agent 既要理解业务又要写代码又要自我评估时的注意力分散。

### 2.2 主流程

系统的运行流程分为三个阶段：

**启动阶段：**

1. 初始化工作目录，加载或创建运行状态
2. 如果存在历史状态，自动恢复到上次中断的位置
3. BusinessAgent 分析数据集，生成业务理解文档（`business_guide.md`）
4. 将业务理解文档注入 ExtractDevAgent 的上下文

**迭代阶段（核心循环）：**

```
ExtractDevAgent 执行一轮（编写/修改代码、运行测试）
        ↓
    保存当前状态
        ↓
Supervisor 检查进度（运行评估、查看历史、分析趋势）
        ↓
    Supervisor 做出决策：
    ├── 任务完成 → 退出循环
    ├── 需要业务分析 → 调用 BusinessAgent 后继续
    └── 给出改进建议 → 反馈给 ExtractDevAgent，继续下一轮
```

**收尾阶段：**

1. 自动提交工作区代码变更
2. 输出运行统计（耗时、Token 消耗、迭代次数）

## 3. 核心机制

### 3.1 Supervisor 监督决策

Supervisor 不是简单的"检查是否完成"，而是一个具备全局视角的决策者。每轮迭代后，Supervisor 会：

- 运行评估命令，获取当前提取准确率
- 阅读 Agent 的操作历史，了解已尝试的策略
- 分析准确率变化趋势，判断是否陷入瓶颈
- 综合以上信息，做出四种决策之一：完成、继续、调整业务理解、给出具体改进建议

Supervisor 支持两种响应解析模式：结构化输出（通过 Pydantic 模型约束决策格式）和文本解析（更灵活的自由文本）。结构化模式在决策可靠性上更优，文本模式在建议表达上更灵活，可根据模型能力选择。

### 3.2 业务理解前置

这是系统的一个关键设计：在 ExtractDevAgent 开始写代码之前，先由 BusinessAgent 分析数据集，生成一份业务理解文档。

这解决了一个常见问题——AI 写代码时容易"只看结构不看语义"。比如面对一个表格，AI 可能正确识别了行列结构，但不理解"合计行应该跳过"、"备注列包含关键分类信息"这类业务规则。BusinessAgent 通过预先分析样本数据，将这些隐含的业务知识显式化，写入指导文档供 ExtractDevAgent 参考。

ExtractDevAgent 在编码过程中也可以随时通过工具调用向 BusinessAgent 提问，获取特定业务问题的解答。

### 3.3 状态管理与断点恢复

提取程序的优化是一个长时间运行的任务（可能持续数十分钟甚至更久），中途可能因网络问题、API 限流、人为中断等原因停止。系统设计了完整的状态持久化机制：

- **SessionManager** 统一管理所有 Agent 的对话历史、迭代计数、Token 统计
- 每轮迭代后自动保存状态到磁盘，采用防抖策略（5 秒内合并多次保存请求）避免频繁 I/O
- 重启时自动检测并加载历史状态，从上次中断处继续
- 恢复时自动清理未完成的工具调用消息，避免 API 报错

这意味着一次运行中断后，不需要从头开始，已有的代码进展和优化历史都会保留。

### 3.4 超时控制

为了控制运行成本和避免 Agent 陷入无效循环，系统实现了三层递进式的超时提醒机制：

- **常规提醒**：运行超过设定时间后，每隔一段时间提醒 Agent 剩余时间，引导其优先处理高价值改进
- **倒数模式**：临近截止时，每轮都提醒，促使 Agent 收尾当前工作
- **强制中断**：超过最终期限后，后台监控协程强制终止所有 Agent 运行

这套机制通过 Hook 注入 Agent 的消息流，Agent 本身不需要感知超时逻辑，只会在消息中看到时间提醒并据此调整行为。

## 4. 技术亮点

### 4.1 监督-执行分离

多数 AI 编码系统采用单 Agent 模式：一个 Agent 既写代码又自我评估。这种模式的问题在于，Agent 容易陷入局部优化——反复修改同一处代码而忽略全局策略。

本系统将"执行"和"监督"分离为独立 Agent，各自拥有独立的上下文和记忆。Supervisor 不参与具体编码，而是站在更高层面审视进展，能够识别出"Agent 已经在同一个问题上尝试了三次不同方案但都没有效果"这类模式，并给出方向性调整建议。

### 4.2 Hook 驱动的质量保障

系统没有在 Agent 的提示词中硬编码"写完代码要检查语法"这类规则，而是通过 Hook 机制在框架层面自动保障代码质量：

- **Ruff Hook**：Agent 每次写入 Python 文件后，自动触发 ruff 静态检查，发现问题后以系统消息形式反馈给 Agent，促使其立即修复
- **超时提醒 Hook**：在 Agent 处理消息前自动注入时间信息，无需修改 Agent 逻辑

这种设计将"质量保障"从 Agent 的认知负担中剥离出来，变成基础设施层面的自动化能力。Agent 可以专注于业务逻辑，质量检查由系统自动完成。

### 4.3 鲁棒性设计

系统在多个层面考虑了运行时的不稳定因素：

- **API 重试**：对网络超时、连接中断等瞬时错误自动重试，采用指数退避策略（1s → 2s → 4s → ...），重试时自动关闭流式输出以避免断流问题
- **状态恢复鲁棒性**：恢复历史状态时，自动检测并清理"半完成"的工具调用消息（Agent 在调用工具过程中被中断的情况），确保恢复后 API 调用不会因消息格式异常而报错
- **Token 统计**：通过模型包装器统一追踪所有 Agent 的 Token 消耗，便于成本监控和异常检测

## 5. 工具体系

ExtractDevAgent 拥有一套可配置的工具栈，覆盖代码开发的完整需求：

- **文件操作**：查看、写入、插入文件内容，写入工具支持行数限制（防止 Agent 一次生成过长代码）
- **Shell 命令**：执行评估命令、运行测试、查看数据
- **计划管理**：PlanNotebook 工具，帮助 Agent 记录和追踪优化计划
- **业务咨询**：随时向 BusinessAgent 提问业务相关问题

工具集可通过配置按需启用或禁用，例如可以选择性开启 VLM 图片提取、NER 正则匹配等专用工具，根据具体任务需求灵活组合。

## 6. 可配置性与扩展性

系统在多个维度提供灵活配置：

- **多模型支持**：通过模型工厂统一适配 Gemini、OpenAI、Anthropic 等主流模型，切换模型只需修改一个配置项
- **Agent 组合可选**：BusinessAgent 可按需启用或禁用，适应不同复杂度的任务
- **运行参数可调**：最大迭代次数、目标准确率、超时时间、API 并发数等均可通过环境变量或命令行参数配置
- **工具集可定制**：根据数据集特点选择性启用提取工具（文本提取、VLM 图片提取、NER 等）

扩展新能力时，只需在对应层面添加：新的 Agent 类型、新的工具函数、新的 Hook、或新的模型适配器，无需修改主流程逻辑。
